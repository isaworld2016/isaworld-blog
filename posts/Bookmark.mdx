초짜티를 살짝 벗은 개발자.

기존에 운영하던 관리자 웹에 '즐겨 찾기' 기능 추가 업무를 요청 받았다.

(주로 잘하고 싶은) 조급한 마음이 들어 일단 코드부터 열기 시작한다.

> #### 손이 먼저 움직였던 과거를 떠올려보면
> ##### 1) 충분히 검토하지 않고 바로 코딩하기 시작한다.
> ##### 2) 선택했던 UI, UX에 대한 다른 아이디어가 개발 중에 불쑥불쑥 튀어나와서 자꾸 번복한다.
> ##### 3) 결과적으로 원하던 기능은 구현되었지만 아쉽고, 찝찝한 감정으로 마무리한다.

## 이번엔 손보다 머리를 먼저!

1. 기존에 구현했던 UI/UX와 레퍼런스를 충분히 뒤적여보고 초안을 결정한다.

2. 해당 아이디어를 바탕으로 피그마로 구현을 한다. 최대한 구체적으로 정리한다.

3. 프로세스를 최종 정리한다.

4. 화면 구현을 시작한다.

5. 화면 검증 후, api 구현을 시작한다.

6. 화면과 api 검증 후, 코드 리팩토링한다.

---

### 1. 기존에 구현했던 UI/UX와 레퍼런스를 충분히 뒤적여보고 초안을 결정한다.

- '즐겨찾기'를 Side에 구현했던 적이 있었고 Top에 구현했던 적이 있었다. 지금 구조에서 Side에 넣기에는 디자인적으로 무리가 있어서 <b>Top에 배치하기로 한다.</b>

- 하단에 '저장하기'와 '취소'를 두어서 저장하지 않으면 DB에 반영되지 않는 UX 기획.

- 버튼으로 '저장하기'를 구현하려다가 네이버의 UI(section을 나누고 텍스트가 버튼 역할을 함)가 지금 modal에 어울리는 것 같아서 디자인 차용.


### 2. 해당 아이디어를 바탕으로 피그마로 구현을 한다. 최대한 구체적으로 정리한다.

- 피그마를 적극적으로 활용해서 마치 고객사가 요구사항을 줬다고 생각하면서 작성했다.

- '이 문서만으로 개발을 할 수 있을 정도'로 기술하자는 목표로 작성했다.

![상황별로 구분한 UI](https://user-images.githubusercontent.com/57585701/204979424-12cfa690-47b8-46d7-9be1-9ec79c8f1e0a.png)
* 등록된 즐겨찾기가 없을 경우, 5개 이하인 경우, 5개 이상인 경우의 UI를 구체적으로 그려보았다.

* '저장하기'와 '취소' 버튼은 하나의 section으로 묶어서 border를 주고 구분을 주었다. 그랬더니 굳이 '버튼'으로 표현되지 않아도 깔끔한 UI가 완성되었다.

![상상한 바를 구체적인 언어로 기술](https://user-images.githubusercontent.com/57585701/204979728-1e57744f-3944-4282-a706-2b0e41cf3b57.png)

* Frame 바깥 쪽 영역에는 '내가 만약 고객사라면 어떻게 기술할 것인가'를 상상하며 구체적으로 요구사항을 정리해봤다.

* 다른 사람이 보고 이해를 해야 하기 때문에 요구사항을 카테고리로 정리했고 중요도에 따라 정렬하였다.

* 색상표나 아이콘도 따로 정리하여 구현자의 번거로움을 줄였다.


### 3. 프로세스를 최종 정리한다.

1. 사용자가 관리자웹에 로그인하면서 가져오는 개인정보 안에 즐겨찾기에 대한 것도 추가한다. 사용자ID(PK) - 메뉴ID
api url ex) /getBookmarkList
SQL ex) SELECT * FROM BOOKMARK WHERE USER_ID ='isaworld'
JSON ex)

2. 상단 Top NavBar 중에서 별 아이콘을 클릭하면 Modal이 나온다. Modal이 펼쳐질 때마다 /getBookmarkList를 호출하여 리스트를 갱신한다.

3. 모달에서 별을 클릭하여 비활성화/활성화를 수정한 후, 하단에 '저장하기'를 누르면 저장 트랜젝션을 타고 DB에 저장된다. (삭제만 가능)

4. 즐겨찾기 추가는 각 메뉴 화면에서 타이틀 옆에 별 아이콘을 클릭하여 추가하거나 삭제할 수 있다. (저장/삭제 가능)


### 4. 화면 구현을 시작한다.

![bookmarkUi](https://user-images.githubusercontent.com/57585701/205220762-78bb99d6-1a1d-4b81-8415-8266d0888070.gif)

기존 프로젝트에서 [Element UI](https://element.eleme.io/#/en-US/component/dropdown)를 활용하고 있어서 UI 구현에 제한이 있었다. 기획과 최대한 비슷하면서 기존 UI 뤼앙스를 헤치지 않는 선에서 맞췄다.

Element UI에서 dropdown으로 구현되고 있었는데 dropdown을 닫는 event가 라이브러리에서 지원하지 않아 '닫기'버튼을 제외했다.


중요한 것은 큰 가닥이지 작은 디테일이 아니므로 일단 기능의 구현을 완성한 후에 디테일을 잡는다.

디테일이란 아이콘을 뭘로 하고 색상을 어떻게 하고 크기가 어떻고 저떻고 하는 style적인 내용을 의미한다.

일단 빠르고 정확하고 크게크게 구현하고나서 디테일은 나중에 얼마든지 잡을 수 있다.
